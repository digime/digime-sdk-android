Ongoing Postbox

Introduction

An Ongoing Postbox allows your app to push data to the user over multiple app sessions without the use of digi.me app after initial consent has been given*.

From developer perspective the authorization process is almost identical to regular authorization. Under the hood we use OAuth 2.0 with JWT and JWS with RSA signing and verification to issue a medium lived, refreshable OAuth token, which is used to re-query user’s data without the need to leave your app.

The SDK handles all of this for you.

Ongoing Postbox is for you if:

  • You need to regularly push data to your user
  • You are using an ongoing push contract

* refreshTokens used to refresh accessTokens do eventually expire (for example - 30 days). When this happens, user will need to be directed back to digi.me app for re-authorization.

Example

Please refer to our Ongoing Postbox example in the Kotlin Example App for more details on data and metadata.

How to use

Create Ongoing Postbox

Simply use the new authorizeOngoingPostbox method on an instance of DMEPushClient:

pushClient.authorizeOngoingPostbox(activity: Activity) { postbox, credentials, error -> 
    // You may now push data to the postbox
}

You may notice that upon completion of this method, the SDK supplies a DMEPostbox which contains postbox identifier and DMEOAuthToken which contains an OAuth token. This is key to restoring access to the Ongoing Postbox and we recommend you store this - you will need it later.

Our recommendation would be to save it to encrypted shared preferences or other encryption supported local storage.

Refreshing Ongoing Postbox

If you have previously obtained user’s consent, and are in possession of both DMEPostbox, and DMEOAuthToken, you can push data to your users without them having to leave your app.

To do this, simply call the following method on a new DMEPushClient instance, where cachedPostbox is the DMEPostbox, and cachedCredentials is theDMEOAuthToken:

pushClient.authorizeOngoingPostbox(activity: Activity, cachedPostbox, cachedCredentials) { postbox, credentials, error -> 
    // You may now push data to the postbox
}

One important thing to note here - the DMEOAuthToken returned in completion may not be the same token you have passed in. This is because the SDK will try to automatically refresh an accessToken using a refreshToken (both of these contained in DMEOAuthToken), generating a new DMEOAuthToken. This means you should replace your old token with the one you receive in completion.

Push Data to Ongoing Postbox

To push data to our Ongoing Postbox, we prepare DMEPushPayload which contains postbox information, as well as some other, and DMEOAuthToken as well, and call method bellow with these parameters:

pushDataToOngoingPostbox(payload: DMEPushPayload? = null, credentials:  DMEOAuthToken? = null) { isDataPushSuccessful,  error  ->
    // Handle error or store updated Ongoing Postbox
}

Completion block will give us back boolean value if push of data was successful or not and error for us to handle in that case.

Configuration Options

There is a new property available on DMEPushConfiguration object - autoRecoverExpiredCredentials. This defaults to true, which means that if the refreshToken contained in DMEOAuthToken has expired, the user will be directed to the digi.me app, so that this can be regenerated.

If you wish to direct the user back to digi.me app manually, set this property to:

configuration.autoRecoverExpiredCredentials = false;

When set to false, the SDK will return an TokenExpired error in completion.
